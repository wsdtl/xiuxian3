# 修仙模块架构和扩展教程

本文按当前代码整理，冲突时以代码为准。`修仙` 是玩法根模块：根目录放公共能力，中文二级包按玩法拆分并挂载 WS 命令。组件开发边界见 `开发约束.md`，本文偏架构和扩展流程。

## 模块边界

- 修仙帮助、修仙百科、后台接口、玩家、背包、纳戒、保险箱、修仙物品、源库、二手市场、贸易服务、宗门、探险、武器、装备、铭刻、对战、异界虫洞、首领、修仙界历史、数据库备份都是中文二级组件。
- HTTP 路由由 `后台接口` 和 `修仙帮助` 承接；`后台接口` 是 web 后台 API 承接点，`修仙帮助` 提供 Hexo/Vexos 风格 Markdown 文档站。
- 数据库使用 sqlite3，schema 版本为 `SCHEMA_VERSION = 2026062001`；`player_weapons` 保存累计经验，但不保存攻击、自带技能或附魔槽位这类可实时派生字段；`sects` 保存宗门山门坐标、创建者和宗主，`sect_members` 保存宗门成员归属，`sect_stats` 保存宗门等级和三底蕴，宗门战数据按周期累计到宗门影响力、个人贡献和奖励表。
- 行为沉淀使用长期表：`game_logs` 记关键行为流水，`player_lifetime_stats` 接清理前的累计统计，`player_journals` 记玩家日记摘要，`player_titles` 记动态称号，`daily_fortunes` 记每日气运，`weapon_legends` 记武器传奇。
- 每个二级包保留 `说明.md`，作为单个组件的使用和扩展约束；帮助站会读取这些 Markdown。
- `稳定版封版记录.md` 保存当前可运营基线、验证结果和上线前检查。
- 常用检查覆盖冒烟测试、WS 触发测试、命令压力测试和架构业务自查。

## 命名规则

- 二级目录统一使用中文，例如 `修仙帮助`、`后台接口`、`玩家`、`源库`、`纳戒`、`贸易服务`。
- Python 文件名、类名、函数名、变量名统一用英文。
- 玩家可见命令和回复文本使用中文。
- 命令入口以当前代码显式注册为准；保留的别名也属于正式入口，不支持未注册命令。

## 生命周期

- `修仙/__init__.py` 注册 `OnEvent.connect(priority=50)`，启动时执行 `db.init()`。
- `修仙/修仙帮助/__init__.py` 注册 `OnEvent.connect(priority=40)`，启动时递归读取 `修仙` 下的 `.md` 文档并缓存帮助站。
- `修仙/修仙百科/__init__.py` 注册 `OnEvent.connect(priority=40)`，在数据库初始化后读取 Markdown 和 `xiuxian.db` 结构化资料并缓存百科知识。
- `修仙/数据库备份/__init__.py` 注册 `OnEvent.disconnect(priority=100)`，关闭时先备份数据库。
- `修仙/__init__.py` 注册 `OnEvent.disconnect(priority=50)`，关闭时释放数据库连接。
- 如果关闭回调按倒序执行，备份优先级应保持高于数据库关闭优先级。

## 消息传播

客户端连接：

```text
/ws/bot/{client_id}
```

标准输入：

```json
{
  "code": 202,
  "type": "text",
  "message": "修仙信息",
  "request_id": "本次请求唯一 id"
}
```

传播流程：

1. WS 驱动收到文本并解析 JSON。
2. 协议校验要求 `code/type/message/request_id` 都存在，正常通讯 `code` 为 `202`。
3. WS 层按 `client_id + request_id` 做短期幂等保护，重复请求直接返回 `code=202`。
4. 驱动按命令匹配触发器，未命中返回 `code=404`。
5. 命中后创建后台任务，并把当前 `request_id` 放入 WS 请求上下文。
6. `WsMessageHandler` 拆出 `cmd` 和 `message`。
7. 精确命令取第一个空格前文本为 `cmd`，空格后原文为 `message`。
8. WS 层会先把 `[CQ:at,qq=xxx]` 转成 `xxx`，再压平多余空格。
9. 修仙命令通常只接收 `client_id` 和 `message`。
10. 业务返回文本后，`send_reply` 发给当前 `client_id`。
11. 文本回复会自动加 `【玩家名·称号 Lv.等级】` 前缀；图片回复不额外加标题。
12. 修仙文本回复都会经 `send_reply` 转成 markdown，并自动补玩家头；按钮按“业务手写、正文预测、当前组件、默认兜底”合并，手写 `<休息>` 这类标记永远优先。

## 参数约定

- `client_id`：玩家唯一 id，来自 WS 路由。
- `message`：触发命令后的业务参数，保持用户原文结构；业务需要结构化参数时，在各自 service 内按规则解析。
- `raw_message`：完整原始文本，仅高级自定义功能需要。
- `message_data`：完整 WS JSON；一般业务不需要读取，`request_id` 由 WS 层自动维护。
- `match`：正则触发时的匹配对象；修仙当前主要使用精确命令。

## Markdown 按钮

- 按钮工具统一放在 `修仙/markdown_utils.py`，二级包不要再复制一份。
- `button("修仙信息")` 默认 `button_type=1`；需要其它按钮行为时再显式传参。
- `MarkdownKeyboard` 协议最多 25 个按钮，每行最多 3 个，超过 25 个会自动截断。
- 修仙回复层保留 15 个硬上限；业务手写按钮页按 15 个上限展示，普通自动补按钮目标为 6 个以内。
- 业务失败、条件不满足、需要告诉用户下一步时，`T.hint()` 会调用 `T.tip()` 生成斜体提示，并把建议原文放到正文末尾。
- 建议文案要写给用户看，例如 `发送：探险列表 查看地点，再发送：导航 地点名`。
- 回复层不从带占位参数的建议文案里猜按钮；没有 `<命令>` 标记时，也会因为预测按钮、当前组件按钮或默认按钮统一作为 `markdown` 发送。
- 想生成按钮时由业务手写 `<命令>`，例如 `血气不足，可以先<休息>`。
- 尖括号里的内容会原样作为按钮命令，是否能点击执行由业务自己控制。
- 需要显示文字和实际命令不同时，使用 `<实际命令:显示文字>`，例如 `<商场推荐:去商场>`。
- 默认按钮来自 `reply.py` 的 `DEFAULT_BUTTONS`，当前为 `指南`、`状态`、`修仙信息`。
- 预测按钮来自回复正文关键词，只生成固定命令，不自动生成需要用户补参数的命令，单次最多先取 3 个。
- 当前组件按钮来自 `reply.py` 的 `CONTEXT_BUTTONS_BY_GROUP`，只在自动补齐阶段使用；预测命中时补足到 6 个以内，没有预测时才按当前组件兜底。
- 默认按钮只在没有手写、预测和当前组件按钮时出现。
- 最终顺序是业务手写按钮、预测按钮、当前组件按钮、默认按钮，并按实际发送命令去重。
- `指南` 只作为方向索引，具体入口下沉到 `指南 成长/行囊/战斗/交易/世界`；探险列表、装备孔位等主动打开的业务入口页可以给完整按钮；普通失败提示、状态回执、结算回执要保持少而准。
- `指南` 和帮助图这类导航页可通过 `auto_buttons=False`、`default_buttons=False` 关闭自动补齐。

## 玩家引用

- 当前正式对外输入优先使用玩家名称，因为新接口 id 很长。
- 涉及对方的命令也可以直接@对方；用户不需要手动输入长 id。
- WS 层会把平台 @ 码转成内部标识，业务层继续按名称或内部标识解析。
- 名称唯一，避免同名玩家导致对战、购买、接受请求时无法分辨。

## 本体和行商化身

- 本体：探险、休息、首领、虫洞、切磋、押注决斗、抢劫。
- 行商化身：跑商、导航、商场买卖、出售分流、二手市场。
- 探险中可以跑商，因为行商化身在外行动。
- 普通跑商出售不直接发放复杂材料，也不能购买恢复药；药路状态只在跑商结算时影响顺药随机事件、补给效率和可顺药上限。
- 当前跑商入口与普通探险地点、世界状态承接城池重合，每个地点 3 个纯经济货物；太虚秘境是秘境，不参与跑商，也不承接普通世界物资回收。
- 纯经济货物价格基准上调，不受每日随机价格波动影响；普通利润收益曲线仍用于约束过量出货。
- 修仙物品设定分成世界物资和角色成长道具。世界物资走纯经济、药路、民生、建设、古物、战利品六大类；武器、技能书、宝石、恢复药、开孔器、洗髓液、淬锋丹、铭刻之羽和源石不进入六大类。
- 纯经济只包含当前 11 个普通跑商地区各 3 个特产，共 33 个；历史主题和其他世界物资大类不能塞进纯经济。
- 药路、民生、建设、古物统一通过 `出售` / `自动出售` 流入 11 个承接城池；战利品通过同一入口流入特殊收购势力，纯经济走商场买卖。
- 恢复药不能通过商场购买，当前主要由探险等玩法产出。
- 探险中不能主动挑战首领、虫洞或玩家对战，因为这些都要求本体空闲；但正在探险中的玩家可以作为抢劫目标，防守方使用探险开始快照。
- 抢劫会产生仇恨；有仇恨关系时，`修仙信息` 和 `状态` 展示当前对自己仇恨最高的死敌和报复指数，报复指数 = 仇恨值 x 20，最高 100。

## 组件联动地图

修仙组件按“输入事实 -> 公共能力 -> 输出沉淀”协作，不靠二级包互相导入：

- 统一回复链路：所有 WS 组件通过 `send_reply` 输出，回复层补玩家头、系统消息队列、个人通知栏、预测按钮、当前组件按钮和默认按钮；业务手写 `<命令>` 按钮保留完整入口。
- 文档知识链路：`修仙帮助` 和 `修仙百科` 启动时读取 Markdown；组件 `说明.md` 不是旁注，而是帮助站、百科综合回答和开发审计的知识源。
- 库存链路：探险、跑商、抢劫、首领、虫洞和藏宝图产出物品；背包/纳戒/武器库/装备/源库分别承接，保险箱提供冻结保护，贸易服务和二手市场处理流转。
- 世界物资链路：纯经济只跑商；药路、民生、建设、古物通过出售进入 11 个城池状态；战利品进入特殊收购和战备蓄能；修仙早报、地图文本、虫洞和宗门增益读取这些状态。
- 战斗链路：探险、切磋、决斗、抢劫、首领和虫洞都走 `combat_core.py`，短摘要走 `combat_log_text.py`，网页明细走 `战斗日志/site.py`。
- 宗门链路：宗门成员行为沉淀影响力、供养和山门建设；城池等级提供范围增益；宗门战奖励生成淬锋丹待领，纳戒负责消耗，武器负责呈现上限变化。
- 反映链路：通知、称号、修仙日记、人物志、风云榜、早报和历史只反映已经发生的业务事实，不反向制造奖励或改写玩法状态。

开发新功能时，先确认它落在哪条链路，再同步更新源组件、承接组件和公共设定。

## 库存边界

背包：

- 保存占负重物品。
- 适合世界物资：纯经济、药路、民生、建设、古物、战利品。
- 有格子上限、负重上限和物品堆叠上限。

纳戒：

- 保存不占负重物品。
- 适合角色成长道具：恢复类、福袋、洗髓液、开孔器、宝石、技能书、铭刻之羽、淬锋丹等。
- 宝石单独按名称和等级存入 `gem_items`。

武器库：

- 武器是独立实例，保存在 `player_weapons`。
- 同名武器用 `武器#ID`、等级、经验、攻击、品质、附魔数区分。
- `player_weapons` 保存累计经验，武器等级由经验和等级上限决定；不保存攻击、自带技能和附魔槽位，攻击由 `common.py` 的 `weapon_attack_value()` 按模板基础攻击、品质、等级和武器类型实时派生，自带技能来自 `weapon_defs.skill_id`，附魔槽位由武器等级实时计算。

保险箱：

- 默认 18 格，保存到 `vault_items` 和 `vault_weapons`。
- 背包、纳戒和宝石存入后会从原库存移走；取出后回到原库存。
- 备用武器存入后会托管到保险箱 owner，取出后回到武器库。
- 保险箱内物品冻结，不能使用、出售、回收、上架、升级或附魔。
- 已装备武器和最后一把武器不能存入保险箱。

装备：

- 创建玩家时默认拥有所有装备位。
- 装备不进入背包或纳戒。

## 数据边界

- 二级包之间不要互相导入。
- 需要跨玩法复用的能力放到根目录公共模块。
- 战斗统一走 `combat_core.py`。
- 武器实例统一走 `weapon_core.py`。
- 玩家、库存、源石、装备加成、天气、灵潮等通用能力统一走 `common.py`。
- 闪避、恢复、探险、跑商、抗暴这类百分比加成在 `common.py` 汇总后统一做收益递减封顶；固定血气、精神、防御仍按数值直接相加。
- 异界虫洞业务放根目录 `wormhole_service.py`，贸易服务只负责尝试发现虫洞。

直接流水清理：

- `combat_logs` 保留 5 天，只保留短战斗流水；网页战斗日志优先读取探险、对战、抢劫、首领挑战和虫洞挑战的结构化结果。
- `trade_prices`、`trade_heat`、`trade_daily_rewards`、`trade_buy_locks`、`daily_fortunes` 保留 7 天。
- `daily_newspapers` 保留 7 天。
- `game_logs`、`trade_records`、`second_hand_records`、三类回收记录、已领取探险记录、已领取虫洞/首领参与记录、虫洞通知、对战记录和抢劫记录保留 7 天。
- 清理每天最多触发一次，入口沿用探险、对战、虫洞、首领等玩法入口的 `cleanup_battle_records()`。
- 每次清理会先删除已不存在玩家的当前态数据，包括背包、纳戒、源库、固定装备、保险箱、称号、日记、未领奖励和非托管武器；历史流水按保留周期处理。
- 第一次启用长期统计时只记录起点，不回填旧记录；之后每次清理会先把到期明细汇总进 `player_lifetime_stats`，再删除明细。
- 未领取探险、未领取虫洞/首领奖励不会被清理误删，等玩家领取后再进入正常清理周期。

## 新增玩法

新增玩法时，新建中文二级包：

```text
灵宠/
  __init__.py
  service.py
  说明.md
```

推荐流程：

1. 在 `__init__.py` 用 `@WsMessageHandler.handler(...)` 注册命令。
2. 在 `service.py` 写业务类，并继承 `CoreService`。
3. 只从根目录公共模块拿共享能力，不导入其他二级包。
4. 如果需要新表，把建表和种子数据写进 `sql.py`。
5. 如果需要新公式，写进 `rules.py`。
6. 如果需要新全局常量，写进 `constants.py`。
7. 给二级包补 `说明.md`。
8. 补冒烟测试或 WS 触发测试。

python -B test\修仙_架构业务自查.py
