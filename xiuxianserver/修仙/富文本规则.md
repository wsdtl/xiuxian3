# 修仙富文本规则

本文只定义修仙组件的回复排版规则，不定义业务逻辑。

## 一、底层支持格式

当前可用的富文本能力：

```markdown
# 一级标题
## 二级标题
**加粗**
_斜体_
**_加粗斜体_**
~~删除线~~
- 无序列表
1. 有序列表
---
> 引用块
```

修仙组件优先使用：

```markdown
**重点文字**
*提示文字*
> **🌱 正文栏目**
> 正文卡片
> 正文卡片内用空行分段
<按钮命令>
```

## 二、消息整体结构

一条修仙回复最多分为四块：

```markdown
【玩家名·称号 LV等级】

> 正文区
> 正文区

*提示区*

<按钮区>
```

规则：

- 消息头由统一回复层生成，不由业务函数手写。
- 消息头格式为：`【玩家名·称号 LV等级】`；`LV等级` 是当前世界皮肤下的等级显示名，默认 `LV1` 到 `LV100`。
- 生成消息头时，统一回复层也会补齐按钮。
- 正文、提示、额外按钮由业务函数按需要生成。
- 业务函数不要自己拼 `【玩家名】`。
- 普通文本回复会统一由发送层转换为 markdown，因为发送层会自动追加按钮。

## 三、发送层判断规则

发送层只做包装，不根据文字内容猜格式。

当前规则：

```text
普通字符串                  -> markdown，自动加玩家头，按正文预测、当前组件、默认兜底补按钮
普通字符串，有 <按钮>        -> markdown，手写按钮在前，必要时少量补预测按钮
dict type=text               -> markdown，自动加玩家头，按正文预测、当前组件、默认兜底补按钮
dict type=markdown           -> markdown，自动加玩家头，按正文预测、当前组件、默认兜底补按钮
dict type=image              -> image，message 只放图片对象
```

格式由业务层自己决定：

```python
return "普通成功文本"
return T.tip("当前血气不足，建议先休息一下")
return T.panel().render()
return battle_log_text
return {"code": 202, "type": "image", "message": image_io}
```

## 四、正文卡

正文卡用于展示资料和详情。

适合使用正文卡：

- 修仙信息
- 武器详情
- 装备详情
- 银行信息
- 人物志
- 背包、纳戒详情
- 商场、二手市场详情

正文卡统一用引用块：

```markdown
> **🌱 状态**
> 经验 **38780/39253**｜血气 **854/886**｜精神 **434/434**
> 原石 **42,617**｜探险中｜自动用药 开启
> 地点：雷泽城 (29,24)
>
> **⚔️ 武器**
> 天下第一盘（乾坤盘）[珍品]
> 攻击 **+110**｜技能：青云斩（破军刺）
```

正文卡规则：

- 正文卡每一行都加 `>`。
- 栏目名统一通过 `MdPanel.section()` 生成，格式为 `**语义图标 栏目名**`；没有命中语义规则时仍保留 `**栏目名**`。
- 关键数字只在详情页必要时局部加粗；`状态` 这类简版面板不要大面积加粗，避免信息层级混乱。
- 大段之间用空行分隔，也就是调用 `panel.hr()` 后只输出一行空引用。
- 正文卡内禁止单独使用 `---`、`===` 这类 Markdown 分割线或标题下划线；当前客户端会把上一行误渲染成大号粗体。
- 展示武器实例编号统一用 `〔12〕`，避开 `#`、`-` 开头等 Markdown 特殊格式。
- 命令输入仍支持 `武器#12`、`#12`、`武器 12`、`12` 等既有写法；展示格式和输入格式分离。
- 不在正文卡里重复展示玩家名、称号、等级。

栏目图标规则：

- 图标只作为正文卡栏目标记，不进入命令、按钮、数据库字段、URL、物品 ID 或任何业务判断。
- 一个栏目最多一个图标；不要在正文每一行都加图标，也不要堆叠装饰。
- 图标由 `修仙/format_text.py` 里的 `section_title()` 集中生成，业务组件不要各自硬编码栏目图标。
- 图标必须能帮助玩家快速识别区域，例如 `🌱 状态`、`📦 背包/纳戒`、`⚔️ 武器/战斗`、`🗺️ 地图/城池`、`🏯 宗门`、`💰 银行/跑商`、`📜 指南/百科/历史`、`✨ 奖励/祈愿`。
- 简短成功文本、失败提示、战斗日志短结算和通知栏不强制加图标；这些内容优先保持短、准、醒目。

## 五、提示

提示用于失败、条件不足、冷却、下一步建议。

提示统一使用单星号斜体：

```markdown
*当前血气不足，建议先休息一下*
```

规则：

- 提示不用 `>`。
- 提示不写 `提示｜xxx` 这类标题。
- 提示放在正文之后、按钮之前。
- 没有必要提示时不要强行加提示。

示例：

```markdown
*当前血气不足，建议先休息一下*

<休息><修仙信息>
```

提示函数分两层：

- `T.tip()`：只负责把一句提示变成斜体，适合极短的单句提示。
- `T.hint()`：负责生成完整提示，会调用 `T.tip()`，并组合建议文字和人工按钮。
- `T.attach()`：负责给正文卡、成功文本、战斗日志补充说明和人工按钮，不会把正文改成斜体提示。

完整提示统一由 `T.hint()` 生成：

```python
T.hint(
    "当前状态已变化，不能休息。",
    "发送：修仙信息 查看当前状态后再操作。",
    buttons=("修仙信息",),
)
```

输出：

```markdown
*当前状态已变化，不能休息。*
发送：修仙信息 查看当前状态后再操作。

<修仙信息>
```

规则：

- `T.hint()` 的业务按钮内容必须来自原业务文本中已经人工写好的 `<...>`。
- 原文没有业务按钮时，不自动新增业务按钮。
- 统一回复层会给普通 text/markdown 回复追加预测按钮、当前组件按钮或默认按钮，但默认按钮只做最后兜底。
- 默认按钮是 `指南`、`状态`、`修仙信息`。
- 手写按钮来自业务文本里的 `<命令>` 或显式传入的 `buttons`，展示顺序永远排在预测按钮、当前组件按钮和默认按钮之前。
- 原文有几个按钮、什么顺序、是否带参数，都必须原样保留；主动打开的入口页可以给完整按钮。
- 回复层只按正文关键词预判固定命令按钮，不从 `发送：xxx 数量` 这类占位建议里生成带参数按钮。
- `suggestion` 是普通建议文字，不负责生成按钮。
- 失败、条件不满足、冷却、重复领取、重复签到等不满足条件的分支用 `T.hint()`。
- 成功结果、状态正文、战斗日志后面追加下一步说明时用 `T.attach()`。

## 六、成功文本

成功文本使用正常文本，不强制加富文本格式。

示例：

```text
签到成功，获得原石 12000。
```

如果成功文本里有重要数字，可以局部加粗：

```markdown
签到成功，获得原石 **12000**。
```

## 七、按钮

按钮永远放在消息最后。

按钮文本使用原有手写格式：

```markdown
<休息><修仙信息>
```

如果展示文字和实际发送命令不同，可以使用冒号格式：

```markdown
<实际命令:显示文字>
<商场推荐:去商场>
```

规则：

- 正文里不要混入按钮。
- 提示后面可以跟按钮。
- 需要按钮时使用统一工具函数生成，不到处手写长串尖括号。
- 人工写过的按钮内容不能乱改，重构时只允许把 `<命令>` 等价改成 `T.buttons("命令")`。
- 已经具备完整参数的按钮必须保留完整命令，例如 `<导航 琢玉楼>`、`<商场出售 星官旧简 3>`。
- 仍需玩家补参数的按钮必须使用 `button_type=2` 并只填入输入框，不直接发送，例如 `建立宗门` 会生成 `建立宗门 x y 宗门名` 的可编辑按钮。
- `<实际命令:显示文字>` 会显示冒号后文本，点击时发送冒号前文本；中文冒号 `：` 也支持。
- 动态按钮可以直接内联 f-string，不需要为了一个按钮单独定义临时变量。
- 回复层最终会按“手写按钮、预测按钮、当前组件按钮、默认按钮”的顺序合并，并按实际发送命令去重。
- 业务手写按钮页最多保留 15 个；普通自动补按钮目标为 6 个以内。
- 预测按钮单次最多先取 3 个，预测命中后可以从当前组件补足到 6 个以内。
- 默认按钮只在没有手写、预测和当前组件按钮时出现。
- 带 `cmd_data` 的按钮按实际发送命令去重，例如 `<商场推荐:去商场>` 和 `<商场推荐>` 只保留前者。
- `CONTEXT_BUTTONS_BY_GROUP` 只负责当前组件按钮；它会过滤掉已经由手写按钮、预测按钮或默认按钮占用的命令。
- `指南` 只保留方向索引，具体入口放到 `指南 成长/行囊/战斗/交易/世界`；探险列表、装备孔位等主动入口页保留完整按钮。
- `指南`、各方向指南和帮助图这类导航页不自动补当前组件按钮和默认按钮，避免按钮过多。

无框命令链接用于系统消息队列和个人通知栏，解决“有待办但不占底部按钮位”的场景。

格式：

```markdown
[展示文字](mqqapi://aio/inlinecmd?command=URL编码后的命令&enter=true&reply=false)
```

规则：

- 无框命令链接只放在短通知、系统消息和极短入口提示里，不替代正文末尾的 `<命令>` 按钮系统。
- 通知链接默认 `enter=true`，点击后自动发送命令；`reply=false`，不引用原消息。
- `command` 必须按 URL 规则编码，避免空格、中文和特殊符号破坏链接。
- 通知对象保留完整 `text` 语义，消息头展示 `label`；没有 `label` 时才退回完整 `text`。
- 个人通知 label 使用中短标签，保留玩法名和动作，去掉 `待领`、`可领`、`待处理` 这类重复尾巴，例如 `首领奖励待领 -> 首领奖励`、`银行结息可领 -> 银行结息`、`今日签到待领 -> 今日签到`。
- 系统消息 label 保留事件类型和地点或核心识别物，不铺完整剧情；普通/战备虫洞优先展示地点，例如 `异界虫洞：陨炉泰坦@玄铁岭 -> 异界虫洞·玄铁岭`。
- 容易误操作的事项只进入安全面板，不自动确认不可逆动作；例如对战请求进入 `决斗记录`，不自动接受。
- 系统消息是世界事实入口，点击后进入对应玩法或状态页；是否能挑战、领取或处理，仍由业务命令自己校验。
- 系统消息和个人通知都不能使用尖括号按钮标记，避免污染业务手写按钮和正文预测按钮。

按钮工具：

```python
T.buttons("休息", "修仙信息")
```

输出：

```markdown
<休息><修仙信息>
```

动态按钮示例：

```python
return T.hint(
    "当前位置可以买入后出售。",
    "建议直接出售当前推荐商品。",
    buttons=(f"商场出售 {options[0]['item_name']} {options[0]['quantity']}",),
)
```

不要为了单次使用写成：

```python
sell_cmd = f"商场出售 {options[0]['item_name']} {options[0]['quantity']}"
return T.hint("当前位置可以买入后出售。", buttons=(sell_cmd,))
```

只有同一个动态命令需要复用多次时，才单独定义变量。

## 八、铭刻展示

铭刻名放在原名之前，格式固定为：

```text
铭刻名（原名）
```

示例：

```markdown
武器：天下第一盘（乾坤盘）[珍品]
技能：青云斩（破军刺）
附魔：青云破（风刃书）
装备：月华沉梦（饰品） Lv10 3孔
```

规则：

- 所有铭刻都使用 `铭刻名（原名）`。
- 展示时只使用 `铭刻名（原名）`，不拆成“原名一行、铭刻一行”。
- 同名但有差异的物品，仍然要显示编号、等级、孔位等区分信息。

## 九、战斗日志

战斗日志不用正文卡模板。

探险、首领、虫洞、切磋、决斗等战斗结果使用统一短结算格式。
消息正文只展示结果、收益、最终状态和战斗日志链接；完整逐回合、技能、掉落和结算明细进入战斗日志网页。

示例：

```markdown
> **探险结束**
> 战斗 **8** 场｜胜 **7**｜败 **1**
> 经验 **+2191**｜武器经验 **+640**｜等级：LV36，未升级
> 最终状态：血气 **0/1136**｜精神 **0/772**
> 战斗日志：[探险战斗日志〔123〕](完整地址)
```

规则：

- 战斗日志可以用 `>`，但不走资料卡的栏目模板。
- 消息正文不展示逐场摘要、行动次数或技能统计。
- 网页战斗日志必须保留逐回合明细、我方和敌方关键参数、技能名；铭刻技能用 `铭刻名（原名）`。
- 不把逐回合完整长日志铺在消息里；需要详细记录时走战斗日志网页。
- 消息里使用 Markdown 改名链接展示网页入口，例如 `[探险战斗日志〔123〕](完整地址)`。
- 所有网页入口都必须隐藏真实地址，只显示 Markdown 改名链接，不在消息正文裸露 `http://` 或 `https://`。
- `战斗日志 开启` 表示网页默认展开逐回合明细；`战斗日志 关闭` 表示网页默认展示摘要。

## 十、统一工具类

当前已经使用统一文本工具类集中管理格式。

位置：

```text
修仙/format_text.py
```

核心接口：

```python
class T:
    """修仙文本格式工具。"""

    @staticmethod
    def tip(text: str) -> str:
        """提示文本，统一使用斜体。"""

    @staticmethod
    def success(text: str) -> str:
        """成功文本，保持普通文本。"""

    @staticmethod
    def hint(text: str, suggestion: str = "", buttons: tuple[str, ...] = ()) -> str:
        """提示文本，可附加建议文字和人工按钮。"""

    @staticmethod
    def attach(text: str, suggestion: str = "", buttons: tuple[str, ...] = ()) -> str:
        """正文或成功文本附加说明和人工按钮。"""

    @staticmethod
    def buttons(*commands: str) -> str:
        """生成按钮串。"""

    @staticmethod
    def panel() -> "MdPanel":
        """生成正文卡。"""
```

`MdPanel` 只负责正文卡：

```python
panel = T.panel()
panel.section("状态")
panel.line("经验 **38780/39253**｜血气 **854/886**")
panel.hr()
panel.section("武器")
panel.line("天下第一盘（乾坤盘）[珍品]")
return panel.render()
```

输出：

```markdown
> **🌱 状态**
> 经验 **38780/39253**｜血气 **854/886**
>
> **⚔️ 武器**
> 天下第一盘（乾坤盘）[珍品]
```

注意：`panel.hr()` 的名字保留为“分段”语义，但实际只输出空行，不输出 Markdown 横线。

## 十一、例外

以下内容不强制套正文卡：

- 图片消息
- 纯普通成功文本
- 简短错误提示
- 战斗日志
- 代码块
- 调试回复

例外仍然要遵守提示和按钮规则；例如简短错误提示优先用 `T.tip()` 或 `T.hint()`。

## 十二、图片消息

图片消息使用 `type=image`。纯图片直接把图片对象放到 `message`：

```python
return {
    "code": 202,
    "type": "image",
    "message": image_io,
}
```

规则：

- `image` 支持 `bytes`、`bytearray`、`memoryview`、`BytesIO`、`Path` 和类文件对象。
- `type=image` 只发纯图片，不承载文字、markdown 或按钮。
- 需要文字说明时，先发 `markdown` 或 `text` 回复，再单独发 `image`。
- 本地生成图、临时图、战报图和帮助速查图优先使用 `type=image`，不要把本地路径塞进 markdown。
- 祈愿演出这类固定公网 GIF 可以例外嵌入同一条 Markdown 结果；必须使用 `/static/...` 公网地址，并声明展示尺寸。

公网静态图片才考虑 QQ Markdown 图片。QQ Markdown 图片需要声明展示尺寸，普通 `![说明](图片地址)` 可能不会渲染。需要在图片说明后写宽高：

```markdown
![插图 #720px #400px](https://example.com/static/help/example.png)
```

图片地址必须是机器人平台可访问的公网地址；本地路径或内网地址只会作为文本或失效图片处理。临时图、本地生成图和帮助速查图优先使用 `type=image`；`帮助` 和 `地图` 里的网页入口使用隐藏 Markdown 链接，不直接暴露公网 URL，也不再把修仙界地图作为消息图片塞进正文。
