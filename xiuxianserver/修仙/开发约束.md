# 修仙开发约束

本文记录 `修仙/` 业务组件的长期维护边界。约束以当前代码为准；文档只沉淀稳定规则，不复制一次性整理步骤、本机路径或环境配置清单。

## 一、模块边界

- `修仙/` 是玩法根包，玩法、命令、数据库和 HTTP 业务都放在这里。
- `修仙/__init__.py` 是玩法根路由入口，负责暴露 `router`，并通过导入二级包触发 WebSocket 命令注册。
- `sql.py`、`common.py`、`rules.py`、`combat_core.py`、`weapon_core.py`、`reply.py`、`markdown_utils.py` 等根目录模块承接跨组件公共能力。
- 中文二级包按玩法拆分；二级组件只负责自己的命令入口和业务逻辑。
- 二级组件之间不互相导入；跨组件复用能力上收到 `修仙/` 根目录公共模块。
- HTTP 路由由 `后台接口` 和 `修仙帮助` 承接；玩家玩法主要通过 WS 命令触发。

## 二、命令规则

- 修仙命令使用 `@WsMessageHandler.handler(...)` 注册。
- 命令入口以当前代码显式注册为准；代码中保留的别名也属于正式入口。
- 不额外兼容未注册的旧口令。
- 正常 WS 消息必须携带 `request_id`，服务端回复要带回同一个 `request_id`。
- 需要对方的命令支持玩家名或平台 `@`；WS 层会把 `[CQ:at,qq=xxx]` 转成内部 id 并压平空格。

## 三、回复和按钮

- 修仙 `text` / `markdown` 回复统一走 `send_reply`。
- 文本回复会自动补玩家头，并默认升级为 markdown。
- 业务手写 `<命令>` 按钮必须保留，尖括号内文本原样作为按钮命令。
- 已有 markdown keyboard 中的完整按钮对象要保留，避免丢失自定义 `cmd_data`。
- 按钮顺序固定为：业务手写按钮、正文预测按钮、当前组件按钮、默认按钮。
- 默认按钮固定为 `指南`、`状态`、`修仙信息`。
- 按钮去重优先按 `action.data`，数量上限由 `markdown_utils.py` 统一处理。

## 四、文档规则

- `architecture.md` 写业务架构和扩展流程。
- `完整设定.md` 写玩法规则和数值口径。
- `富文本规则.md` 写文本、提示、按钮和正文卡规则。
- 每个二级组件保留 `说明.md`，并在 `## 命令` 下维护可由帮助站读取的命令代码块。
- 帮助页启动时递归读取 `修仙/` 下所有 Markdown 并缓存；修改 Markdown 后需要重启项目刷新帮助页缓存。
- 新增或调整命令、按钮、数值、数据库种子时，同步更新对应组件说明和总设定。
- Markdown 文档内容和代码冲突时，以代码为准，同时更新对应文档。

## 五、验证要求

修改修仙组件、命令、回复、按钮或文档索引后，按影响范围运行：

```powershell
python -B test\修仙_架构业务自查.py
python -B test\修仙_markdown按钮测试.py
python -B test\修仙_冒烟测试.py
python -B test\修仙_ws触发测试.py
python -B test\修仙_命令压力测试.py
```

只改 Markdown 时，至少确认阶段性整理口吻没有回流，并确认组件 `说明.md` 的命令代码块仍覆盖当前注册命令。
