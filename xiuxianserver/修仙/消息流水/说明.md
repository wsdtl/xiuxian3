# 消息流水组件说明

消息流水组件负责把驱动器收到的消息和驱动器发出的回复整理成聊天框式短期流水。它只展示数据流，不参与命令匹配、奖励结算、通知生成或任何玩法状态修改。页面对所有人公开，直接打开即可查看最近流水。

## 概念

- `client_id` 是驱动器传入的原始入口身份。
- `player_id` 是消息流水组件按用户组解析出的主用户身份。
- `flow_id` 是消息流水组件生成的递增序号，用于页面排序和实时推送游标。
- `created_at` 是修仙服务观察到这条消息并写入流水的时间，不是 QQ、WS 或其它协议原始时间。
- `direction` 分为 `incoming` 和 `outgoing`，页面按左收右发展示。
- `adapter` 记录消息来自哪个驱动器，例如 `qq`、`ws`。
- `sender_name` 是页面显示用的发送人名字，收到消息优先展示角色名，发出消息固定展示 `修仙服务`。

## 命令

```text
消息流水
消息记录
```

命令返回消息流水后台入口。真实网页地址必须使用 Markdown 改名链接，不在消息正文裸露。

## HTTP

```text
GET /xiuxian/message-flow
GET /xiuxian/message-flow/api/recent
GET /xiuxian/message-flow/stream
```

页面、历史接口和实时流都直接公开。它展示最近的短期流水，不要求用户组后台登录。

前端页面文件放在 `static/message-flow/`，路由负责读取静态页面模板、注入接口配置和输出公开页面。

## 数据表

`message_flows` 是短期展示表，字段如下：

```text
flow_id
direction
adapter
request_id
client_id
player_id
sender_name
message_type
content
created_at
```

保留规则：

- 数据库最多保留 2 天。
- 数据库最多保留最新 5000 条。
- 内存全局环形缓冲保留 3000 条。
- 单个主用户内存环形缓冲保留 300 条。

## 展示规则

- 收到的消息展示原始可读正文。
- 发出的消息展示业务正文，不展示玩家头。
- 发出的消息不展示个人通知栏。
- 发出的消息不展示系统消息栏。
- 按钮、keyboard 和其它交互元数据不进入流水展示。
- Markdown 正文保留链接、图片、引用块、加粗和斜体等富文本效果。
- 图片消息优先展示可访问链接；无法展示二进制时显示 `〔图片〕`。
- 超长内容只保留前段展示文本，不保存二进制原文或协议大包。

## 组件关联

- QQ 和 WS 驱动器通过 `launch/message_events.py` 发布收发事件。
- 消息流水组件订阅事件后自行解析用户组主用户，不要求驱动器理解修仙业务。
- 用户组组件提供后台 session 和 `client_id -> player_id` 映射。
- 回复层仍然负责玩家头、通知和按钮；消息流水只在展示前剥离这些辅助内容。
- 帮助站和修仙百科会读取本文档，把消息流水纳入项目知识源。

## 维护边界

- 新驱动器只需要发布 `MessageEvent`，不直接导入消息流水组件。
- 消息流水不能反向触发命令、领取奖励或清除通知。
- 协议审计、原始 payload 排查和安全日志不由本组件承担，仍看驱动器日志。
- 如果以后需要跨主用户查看流水，必须另加主人权限入口；当前公开页只看全局短期流，不按 session 传参做个人隔离。
